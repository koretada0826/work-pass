## 2. フェーズ戦略（ここが本プロジェクトの肝）

「先に人材を貯める→揃ったら企業開拓」を、そのままシステムのリリース段階に落とす。

| Phase | 目的 | 作るもの | 完了の目安 |
|---|---|---|---|
| **P1** | 人材を貯める | 人材DB（登録・一覧・詳細・検索）＋管理画面 | 登録が回り、母集団が貯まり始める |
| **P2** | 企業とマッチング | 企業求人DB＋マッチングエンジン＋おすすめ表示 | 人材が一定数貯まった時点で起動 |
| **P3** | 開拓を自律化 | 企業開拓エージェント（監視→提案→人が承認） | P1の母集団が「開拓に値する」規模・構成になった時 |

**設計上の重要ポイント**：P2/P3 で必要になるデータ項目（適性点数・価値観・キャリア志向など）は、**P1の時点で人材DBに“最初から”設計・収集しておく**。後付けだと過去の登録者データが欠損して使い物にならない。→ **P1のデータ設計＝プロジェクト全体の成否を握る。**

## 3. システム構成（技術スタック：作りやすさ優先で選定）

実際に動く／私が構築・保守しやすい／無料枠で始められる、を基準に選定。

```
[ブラウザ]
   │
   ▼
[Next.js 14 (App Router) + TypeScript]  ← 画面＋API を1プロジェクトで完結
   │
   ├── Prisma (ORM)
   ▼
[PostgreSQL]  ← Supabase(ホスト型/無料枠)。ローカルはSQLiteでも可
   │
   ▼
[マッチングエンジン (TypeScript)]  ← 決定論スコア（安定・無料・説明可能）
   │
   └── [Claude API (Opus/Sonnet)]  ← 上位候補だけLLMで解説・微調整（コスト最小化）

[企業開拓エージェント] = cron/スケジュール実行 → DB監視 → Claude API → 提案を人へ通知
```

**選定理由（要点）**
- **Next.js 1本**：フォーム・DB保存・マッチング・管理画面まで1つのコードベースで完結。分離構成より速く・壊れにくい。
- **PostgreSQL(Supabase)**：無料枠・認証・ホスティング込み。件数が増えても耐える。ローカル検証はSQLiteに切替可能。
- **AIは2層**：まず**決定論スコアで足切り**（速く・タダ・理由が明確）→ **上位だけClaude APIで解説**。全件LLMはコストが跳ねるので避ける（第6章）。
- **ホスティング**：Vercel（Next.js）＋Supabase（DB）。両方無料枠から。

## 4. データモデル：人材DB（Phase 1 の心臓部）

企画書の項目を、マッチング計算で使える形（**数値・列挙型**中心）に正規化する。
凡例：`種別` = 入力形式 / **★=マッチングで使う主要項目**

### 4-1. 基本情報 `candidate`
| 項目 | 種別 | 備考 |
|---|---|---|
| 氏名 | text | |
| 年齢 | int | |
| 最寄駅 | text | |
| 通勤可能範囲 | int(分) | ★ 勤務地マッチに使用 |
| 連絡先(メール/電話) | text | |
| 登録日 | datetime | 自動 |
| ステータス | enum(登録済/診断済/面談済/採用/保留/紹介可) | 運用管理用 |

### 4-2. 勤務条件 `preference` ★（勤務条件一致率に直結）
| 項目 | 種別 |
|---|---|
| 希望勤務地（エリア） | text/enum |
| 勤務可能曜日 | 複数選択(月〜日) |
| 勤務可能時間帯 | 開始/終了(時) |
| 希望雇用形態 | enum(正社員/契約/アルバイト/業務委託) |
| 希望年収 | int |
| 希望月収 | int |
| 転職希望時期 | enum(すぐ/3ヶ月/半年/1年/未定) |

### 4-3. 職歴 `work_history`（複数行）
業界 / 職種 / 勤務年数(int) / 実績(text) / 退職理由(text)

### 4-4. スキル `skill` ★
営業・接客・事務・PC・AIツール を各 **0〜5**、保有資格（複数）

### 4-5. 適性診断 `aptitude` ★（0〜100点）
営業適性 / 接客適性 / 事務適性 / マネジメント適性 / コミュニケーション能力

### 4-6. 一般常識テスト `knowledge_test` ★（0〜100点）
ビジネスマナー / 敬語 / 国語 / 計算 / ITリテラシー

### 4-7. キャリア志向 `career` ★
希望職種 / 希望業界 / 3年後目標 / 5年後目標 / 将来やりたい仕事（テキスト＋タグ）

### 4-8. 価値観 `values` ★（各0〜5でウェイト付け）
年収重視 / 成長重視 / 安定重視 / 人間関係重視 / ワークライフバランス重視

### 4-9. 面談評価 `interview` ★（採用担当が入力）
第一印象 / コミュニケーション / 積極性 / 人柄（各1〜5）＋コメント(text)＋評価者・評価日

## 6. AIマッチング設計（中核）

### 6-1. 3層アーキテクチャ
```
第1層：ハードフィルタ（ゲート）
   必須資格・必要経験・雇用形態・勤務地圏内 を満たさない組み合わせを除外
        ↓ 通過分のみ
第2層：決定論スコア（0〜100点）※ここがメイン。速い・無料・説明可能
        ↓ 上位N件のみ
第3層：LLM解説（Claude API）
   「なぜ合うか／懸念点／面談で確認すべき点」を自然文で生成。並びの最終微調整
```

### 6-2. 決定論スコア（第2層）
企画書の5軸をそのまま採点軸にする。各軸0〜100で算出し、重み付き合計。

| 軸 | 中身 | 既定ウェイト |
|---|---|---|
| ① 勤務条件一致 | 勤務地/曜日/時間/雇用形態/給与の合致 | 0.25 |
| ② 適性一致 | 人材の適性点 vs 求める適性基準 | 0.25 |
| ③ 性格一致 | 人材の価値観・面談評価 vs 求める人物像★ | 0.20 |
| ④ キャリア志向一致 | 希望職種/業界/目標 vs 求人 | 0.15 |
| ⑤ スキル・常識一致 | スキル/一般常識テスト vs 必要スキル | 0.15 |

`総合スコア = Σ(軸スコア × ウェイト)`（ウェイトは管理画面から調整可能に）

**各軸の考え方（例）**
- ① 給与：希望年収 ≤ 求人上限 なら満点、下回る度合いで減点。勤務地：通勤可能範囲(分)内なら加点。
- ② 適性：`min(100, 人材点 / 基準点 × 100)` を各項目平均。
- ③ 人物像：人材の該当指標(1〜5) と 求める★(1〜5) の差の小ささを点数化。
- ④⑤ タグ・キーワード一致率＋数値下限クリア率。

→ **すべて理由が数字で説明できる**（「勤務地×、適性92点、給与希望超過」など）。ブラックボックスにしない。

### 6-3. LLM解説（第3層／Claude API）
- 対象：スコア上位のみ（コスト抑制）。
- 出力：`推薦理由` / `懸念点` / `面談確認ポイント` の3点を構造化(JSON)で返す。
- モデル：品質重視の解説は Opus、量産は Sonnet でコスト最適化。

### 6-4. 双方向表示
- 企業向け：求人に対する**おすすめ人材**ランキング＋スコア内訳
- 求職者向け：本人に対する**おすすめ企業**ランキング（※本人希望フラグがある人のみ＝運用フロー⑥Bに準拠）

## 8. 【回答】「人が集まり次第、企業開拓」を自律エージェント化できるか？

### 8-1. 結論
**「全自動の営業ロボ」は非推奨。「Human-in-the-loop（人が最終承認する）自律エージェント」なら実現可能で、しかも相性が良い。**

理由：企業開拓＝営業プロセスは、**判断・分析の部分は自律化に向くが、実際の契約・信頼構築は人間必須**。無差別な自動アプローチは法（特電法・スパム）・レピュテーションのリスクが大きい。よって「エージェントが考えて提案し、人がGoを押す」形が最適。

### 8-2. 企業開拓を5工程に分解して自律化度を判定

| 工程 | 内容 | 自律化度 | 実装方法 |
|---|---|---|---|
| ① タイミング判断 | 人材在庫が開拓に足るか | **◎ 完全自律** | DBを定期監視。件数＋構成（職種/エリア別）が閾値超えで発火 |
| ② ターゲティング | 貯まった人材に“合う”企業タイプを特定 | **○ 自律可** | Claudeで人材プールをクラスタ分析→「今狙うべき業種/職種/エリア」を提示 |
| ③ リスト生成 | 候補企業の抽出 | **△ 半自律** | Web検索/企業リスト/**TELEMO連携**で候補抽出（データ源が必要） |
| ④ アプローチ | 連絡・提案文作成 | **△ 半自律** | 提案文ドラフトは自動生成。**送信前に人が承認**（推奨） |
| ⑤ クロージング | 商談・契約 | **✕ 人間必須** | 人／TELEMOが担当 |

### 8-3. 推奨する動き（P3エージェントの挙動）
```
[毎日/毎週] 人材DBを監視
   ↓
在庫が閾値超え & 特定クラスタ(例:営業適性80+が20人)が形成された
   ↓
Claudeが分析：
   「今、[人材紙工/不動産営業]系の求人を開拓すべき。
     理由：営業適性80+が22名・即勤務可12名滞留。
     推奨ターゲット企業タイプ：__。提案文ドラフト：__」
   ↓
[人へ通知]（ダッシュボード/メール/Slack）
   ↓
人が承認 → TELEMOで架電・開拓（＝既存の会社の武器と接続）
```

これは Ever社の構想（**TELEMOで企業開拓 × WORK PASSで人材蓄積**）と完全に噛み合う。エージェントは「**誰に・なぜ・何を持って架けるべきか**」を吐き出し、TELEMOが実行する。

### 8-4. 技術的にどう作るか
- **MVP**：バックエンドの**スケジュール実行（cron）**が定期的に集計SQLを走らせ、閾値超えたら Claude API に人材サマリを渡し、提案JSONを生成→通知。数十行規模で作れる。
- **段階拡張**：③リスト生成に外部データ源（企業リスト・TELEMOのデータ）を接続していくと精度が上がる。
- **注意**：④の自動送信はやらない（承認ゲートを必ず挟む）。まず「提案まで自律・実行は人」で始め、信頼できたら自動化範囲を広げる。

### 8-5. 一言でいうと
> **「開拓を自動化」ではなく「開拓の意思決定を自動化」する。** 実行（架電・商談）は人／TELEMOに残す。これが安全・現実的・かつ効果が最大。

## 10. 弱点・リスク評価（厳しめ・先に潰す）

1. **P1のデータ設計ミスは致命傷**：後からマッチング項目を足すと過去登録者が欠損。→ 4章の数値項目を**最初から全部**取る。埋まらない項目は「未入力可」でも列は用意。
2. **入力負荷が高い**：項目が多く登録離脱が起きる。→ フォームをセクション分割・一部は面談時に担当が代理入力。適性/常識テストは別画面で。
3. **マッチング精度は初期は出ない**：母集団も求人も少ないと機能しない。→ P1で母集団を貯める戦略は正しい。焦って企業を先に入れない。
4. **自律エージェントの暴走・誤送信リスク**：→ 8章の通り**承認ゲート必須**。送信は自動化しない。
5. **個人情報保護**：人材の機微情報を大量保持。→ アクセス制御・本人同意（グループ会社紹介は本人希望フラグ）・保管方針を運用開始前に明文化。
6. **「相性」を数値化しきれない部分**：→ だからこそ第3層LLM解説と面談評価を残す。スコアは足切り＋優先順位付けに使い、最終判断は人。
